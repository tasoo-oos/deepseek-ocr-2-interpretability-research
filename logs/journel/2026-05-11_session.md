## Survey Findings

- The full fixed survey completed with 1651 successful pages and 0 failures.
- Mean page time was about 17.54 seconds.
- Runtime was dominated by generation length, not activation hook overhead.
- 14 pages hit the 8192 token generation cap.
- Slow pages were mostly dense newspaper or similar high-content pages.
- Runtime correlated almost perfectly with decoder call count.
- Visual and encoder activation magnitudes were moderately higher on generation-limit pages, but this is descriptive only and not causal evidence.
- The high-level `model.infer()` path is CPU/Python-bound on this machine, using about one CPU core and underutilizing the RTX 4090.

## Code Conventions

- Use `uv` for all Python dependency and command execution.
- Keep changes small and direct.
- Keep upstream DeepSeek code under `upstream-ocr-2/` unmodified unless explicitly asked.
- Prefer config-driven scripts over hardcoded experiment settings.
- Keep default configs in interpretability mode with eager attention unless a task explicitly requires speed.
- Do not add compatibility layers unless there is a concrete need.
- Avoid storing full activation tensors for broad surveys; prefer JSONL summaries unless a targeted experiment requires tensors.
- Verify Python changes with `uv run ruff check src scripts` and `PYTHONPATH=src uv run python -m compileall src scripts`.

## Documentation Conventions

- Keep `README.md` practical and user-facing.
- Keep `AGENTS.md` operational and agent-facing.
- Record long experiment context and findings in Markdown under `logs/` or `logs/journel/`.
- Do not use docs to claim mechanistic findings before controlled interventions are validated.
- When adding workflows, document the command, output location, and what must not be committed.
- Prefer concise sections with commands that can be copied and rerun from the repo root.

## Log Conventions

- Use `logs/journel/YYYY-MM-DD_session.md` for human session journals.
- Use `logs/<timestamp>_<name>.md` for generated experiment summaries.
- Use `logs/*.out` for detached command stdout/stderr.
- Use `logs/*.pid` for detached process PID files.
- `.out` and `.pid` files are ignored; Markdown summaries can be committed when they are useful and small.
- Raw run artifacts belong under `outputs/runs/` and should not be committed.
- Raw datasets belong under `data/raw/` and should not be committed.

## How To Read Logs

- For active detached jobs, read the PID from `logs/*.pid` and check it with `ps -p PID -o pid,ppid,stat,pcpu,pmem,etime,cmd`.
- For GPU state, use `nvidia-smi`.
- For command output, inspect the relevant `logs/*.out` file.
- For completed survey findings, start with the generated `logs/<timestamp>_<name>.md` report.
- For detailed per-page survey records, read `outputs/runs/<survey_name>/activation_summary*.jsonl` locally.

## How To Write Logs

- Write detached job output to `logs/<name>.out`.
- Write detached job PID to `logs/<name>.pid`.
- Write durable findings to Markdown, not `.out` files.
- Include enough context to reproduce the run: command, output directory, dataset path, model/config notes, and verification status.
- Keep Markdown logs descriptive unless an experiment has causal controls.

## How To Update Docs

- Update `README.md` when a human user needs a workflow or setup command.
- Update `AGENTS.md` when future agents need operating rules, caveats, or known machine behavior.
- Update `data/README.md` when dataset location or download policy changes.
- Add a journal entry under `logs/journel/` at session boundaries if the session changed workflows, conventions, or experiment state.
- Keep generated run outputs and raw data out of docs commits unless the user explicitly asks for a small tracked report.
